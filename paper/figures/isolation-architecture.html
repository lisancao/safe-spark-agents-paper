Per-tenant isolation, demonstrated layer by layer

Five gates stand between an agent and another tenant's data. On a live EKS cluster, an agent authenticated as tenant A is confined to tenant A at every layer: it cannot connect to, be handed a credential for, execute alongside, ask the catalog for, or reach at storage any of tenant B. Each layer below is not designed-only; it runs, with a captured proof. (Each link is proven independently; a single request traversing all five is the remaining composition step.)

1 · Ingress routing
Envoy · mTLS · routes by client-cert URI-SAN
Demonstrated
proof_2026-07-10_ingress_routing.log
Envoy gateway: terminate mTLS, derive principal from the cert, route on it
A tenant_a cert reaches only spark-connect-tenant-a; tenant_b reaches only its server. An un-granted principal gets 403 (no route) on a plain request (over gRPC, a DENIED reply); a connection with no client cert is refused at TLS. There is no route that can send A's cert to B's server.
2 · Token custody
one Spark Connect server per tenant · injects that tenant's catalog token server-side
Demonstrated
proof_2026-07-10_multiserver.log
spark-connect-tenant-a token: A
The agent never holds the token; the server injects it. When this session reached for tenant_b it was refused: NotAuthorized: Missing Authorization Header (no tenant_b credential to present).
spark-connect-tenant-b token: B
Symmetric by construction: this server holds only tenant_b's token, so a session here cannot present tenant_a's credential.
3 · Execution isolation
each tenant's driver owns its own executor pods
Demonstrated
disjoint pod sets · Spark UI
tenant_a executor pods
2 pods, distinct IPs, owned by A's driver app.
tenant_b executor pods
Separate driver app + pod. The two tenants' compute never shares a JVM.
4 · Catalog authorization
Lakekeeper + OpenFGA + OIDC · per-principal grants
Demonstrated
proof_2026-07-10_perprincipal_authz.log
Shared governed catalog that gates per principal
Each identity is granted only its own warehouse. A tenant_a identity asking for tenant_b's warehouse, table, or vended credential gets 404 (existence hidden); admin and tenant_b get 200; no token is 401. Granting a describe relation toggles the 404↔200, proving the deny is authorization.
5 · Storage scoping
vended, prefix-scoped STS credentials · keyless (IRSA → downscoping role)
Demonstrated
CloudTrail · fleet IRSA = 0 data calls
S3: the final backstop, enforced by the credential itself
Every warehouse read/write runs under the tenant-scoped vended session (CloudTrail: the fleet role makes zero data calls). Replaying that credential across the line is AccessDenied, both directions; an ablation shows a whole-bucket credential would cross, so the deny is the downscoping vend.

How to read this

What remains frontier: multi-tenant scale, many tenants and node autoscaling. Two tenants prove the mechanism; scaling it out is the open problem. Everything else on this page is built and runs, with a captured proof per layer.
