# Catalog dimension: investigation findings (2026-07-16)

5-lens investigation (web + repo) of the governed catalog at fleet scale, for the locked UC-native (UCSingleCatalog + external Delta, Spark 4.1) binding. One facet (catalog-service-scale) failed schema; covered by the others.

## Catalog thesis (sharpened)

The paper's tenancy decision (Model C: a few shared, heterogeneous, multi-cloud Connect servers with per-request credential vend) forces a claim that has been asserted but never grounded: at fleet scale the bottleneck leaves Connect and lands on the governed catalog. That claim is true, but the honest version is sharper than the one in PAPER.md L566. On our locked binding (Unity Catalog OSS 0.5.0 native plugin, UCSingleCatalog over external Delta tables on Spark 4.1), the catalog is deliberately NOT in the commit path: concurrent-writer correctness is Delta optimistic concurrency control resolved at the _delta_log, with atomicity supplied by the storage LogStore, so raw write contention is a Delta-plus-S3 property, not a catalog one. What genuinely IS on the catalog for our binding, and what actually moves off Connect, is the metadata, authorization, and credential-vend path (loadTable, per-principal grant check, sts:AssumeRole per job) plus the structural need for federation across a multi-cloud pool. So the catalog is the real multi-tenant frontier for three concrete reasons: (1) every job funnels through a single-process JCasbin authorizer that is non-thread-safe under concurrent policy reload and a single Postgres metastore; (2) per-request vend collides with the AWS STS account-region request-rate ceiling, which cross-account vending concentrates onto one account; and (3) a pool spanning AWS, on-prem, and Databricks is physically multiple catalogs, so a tenant's identity, grants, and namespace must stay consistent across substrates that UC OSS has no mature federation story for. The catalog-as-commit-bottleneck only materializes if we later adopt UC managed tables plus Coordinated Commits to buy multi-table atomicity, which would validate the blanket thesis by centralizing commit serialization at the catalog. The frontier is real; it is just metadata, authz, vend, and federation today, and commit serialization only under a binding we have not chosen.

## The model

CONCURRENCY. On UC OSS 0.5.0 + external Delta, UC registers a table as a pointer to an S3 prefix; the commit runs through Delta's own transaction log and LogStore, so UC is a pure metastore for concurrency purposes. Delta uses three-phase optimistic concurrency: read the latest snapshot, write data files, then validate-and-commit by trying to create version N+1 of the log. Conflicts surface only at commit as typed exceptions (ConcurrentAppendException, ConcurrentDeleteReadException/ConcurrentDeleteDeleteException, MetadataChangedException, ProtocolChangedException, ConcurrentTransactionException). Two regimes decide fleet behavior, set by TABLE TOPOLOGY, not by the number of Connect servers. Disjoint-table regime (the good case Model C's per-tenant layout buys for free): agents writing their own tenant's tables share no _delta_log and never conflict at any N; the ceiling is S3 request rate on the hot log prefix (about 3,500 PUT/s per prefix) and metadata read amplification as logs grow. Hotspot-table regime (the failure mode): one shared table (audit sink, common dimension, leaderboard) with N appenders degrades toward serial, because only one writer wins each version and the losers re-read a now-longer log and retry, a thundering herd whose per-retry cost rises with both N and log length. Delta append-append is treacherous because two logically disjoint appends into the SAME partition still raise ConcurrentAppendException unless partition predicates are passed, so the mitigations are partition-plus-table-per-tenant (disjoint conflict domains) and platform-owned retry/backoff.

THE MULTI-JVM FLOOR (the Model C correctness prerequisite). Delta's per-table OCC only works if "create version N.json" is atomic. The default S3SingleDriverLogStore serializes writers only WITHIN one JVM. A few shared Connect servers are several JVMs, so two servers writing the same tenant's external Delta table on S3 with the default LogStore and no multi-writer arbiter do not get a clean conflict; they can both believe they wrote version N and silently lose a commit or corrupt the log. Multi-writer-across-JVMs on S3 requires the S3DynamoDBLogStore (DynamoDB conditional PutItem arbitrates the winner) or the emerging S3 conditional-write LogStore (S3 added If-None-Match in Aug 2024; delta-rs uses it since 0.23.0, but the Delta-Spark JVM conditional-write LogStore is still open, delta-io/delta#3596). No Delta LogStore is configured anywhere in the repo (every DynamoDB reference is Terraform state locking), so today's shipped config is the unsafe default. Model C therefore couples tenancy to the concurrency substrate: shared servers REQUIRE a multi-JVM-safe LogStore before the substrate is even correct, before it is fast.

FEDERATION. Model C is a pool spanning AWS, on-prem, and Databricks, which is physically multiple catalogs or catalog instances, not one. A tenant's identity, grants, and table namespace must stay consistent across substrates, and the credential plane is heterogeneous: the AWS leg vends via sts:AssumeRole, an on-prem or Databricks member vends via a different mechanism (Azure SAS, GCP tokens, or UC's own vend). UC OSS has no mature foreign-catalog/federation story (Lakehouse Federation is a Databricks-managed feature). At scale this breaks as divergent grant state across catalogs, no single per-tenant view, cross-catalog vend trust, and split-brain authorization. This is the largest unexamined hole because the multi-cloud pool makes federation structurally mandatory, not optional.

SCALE OF AUTHZ AND VEND. L4: UC authorizes via JCasbin, an embedded RBAC library whose whole model loads into the UC server JVM; Enforce() is cheap, but the model is O(principals x securables x grants) in one process with no sharding, and the default Enforcer is not thread-safe (SyncedEnforcer serializes reads against reloads). So the contention point is the MANAGEMENT plane: every tenant onboard or grant change is a policy reload that must serialize against in-flight checks. The authz decision is verified to fire before any STS call (proof log Part A: cross-tenant vend returns 403, no AssumeRole attempted). Lakekeeper's OpenFGA is a Zanzibar-lineage scale-out service (tuple store, caching, tunable consistency; documented near 1M RPS), so the L4 choice is embedded-library-simplicity vs scale-out-authz-plane, not secure vs insecure; OpenFGA's counter-frontier is that MINIMIZE_LATENCY can serve a stale ALLOW just after a revoke. L5: both catalogs mint short-lived creds by sts:AssumeRole narrowed by an inline session policy to the tenant prefix and pinned with sts:ExternalId (confused-deputy defense), TTL 1h. The hard ceiling is the AWS STS default request-rate quota per account per region, shared across STS operations. Model C's "no standing credential = vend per request" posture is the safest custody stance and simultaneously the worst for this ceiling; at thousands of agents the two collide. Cross-account vend concentrates the entire fleet's quota onto ONE account, because only the account making the AssumeRole request counts against the quota, so a custodian vending cross-account into tenant accounts funnels every tenant through a single chokepoint regardless of tenant count (regional STS endpoints get separate quotas; a support-ticket increase is possible).

IDENTITY GRANULARITY. UC OSS maps the OIDC email claim to a pre-provisioned UC user with no JIT principal provisioning (JIT is only for the credential). So catalog identity is tenant-grained and static while orchestration identity is agent-grained and ephemeral. The realistic fleet shape is coarse pre-provisioned TENANT principals in the catalog plus fine-grained per-agent identity pinned above it (our PrincipalPinningInterceptor + sandbox_<principal> convention), with the agent-to-tenant map enforced outside UC.

THE SECTION-4 HANDOFF (why custody is load-bearing, not plumbing). Omnigent custody is the natural cache/coalesce point: holding the vended credential per (principal, prefix) and reusing it within TTL turns STS load from linear-in-requests into linear-in-active-(principal, prefix, TTL-window)-tuples, the single biggest lever on the quota, and coalesces concurrent same-prefix requests into one vend. Custody also is where platform-owned retry/backoff for hot Delta tables belongs, which matters precisely because untrusted agents supply no backoff discipline and would otherwise turn a hotspot into an S3 503 SlowDown storm on the shared log prefix that spreads to unrelated writers. This ties concurrency back to the paper's custody thesis: because SDP agents emit only inert declarative transforms and never agent code (Section 2), the platform, not the agent, can and must own commit, retry, and vend-cache policy. But custody RELIEVES the STS ceiling only by CONCENTRATING three things: the credential material (one high-value store of live downscoped creds), the principal-to-credential binding (moved off the proven catalog gate onto the custodian), and any cross-account cross-substrate quota chokepoint. The safest posture (never standing) is the worst for the quota, and the fix (cache to TTL) reintroduces a bounded standing credential and concentrates trust. That exact tension is the §3-to-§4 seam.

## Paper-ready draft (§3.3.x)

## 3.3.x The governed catalog at fleet scale: the real multi-tenant frontier

The tenancy decision (§3.3, Model C: a few shared Connect servers, heterogeneous and possibly multi-cloud, each session handed a credential vended per request) relocates the scaling question. With a shared pool the number of Connect servers is no longer what a fleet of agents presses on; the pressure lands on the governed catalog and the credential plane behind it. This subsection states, for the binding we actually locked (Unity Catalog OSS 0.5.0 native plugin, UCSingleCatalog over external Delta tables on Spark 4.1), what is a stable design property today and what is honestly still the frontier. Every catalog proof in this paper is two-tenant, single-threaded, and sequential; nothing below that is labelled a measurement has been measured.

### The commit model is a property of the (catalog, table-type) binding, not of "the catalog"

On UC OSS with external Delta tables, the catalog registers a table as a pointer to an S3 prefix and the commit runs through Delta's own transaction log, not through the catalog. For concurrency the catalog is a pure metastore. Delta uses optimistic concurrency control: a writer reads the latest snapshot, writes data files, then validates and commits by creating the next version of the log; conflicts surface only at commit as typed exceptions (ConcurrentAppendException, ConcurrentDeleteReadException, ConcurrentDeleteDeleteException, MetadataChangedException, ProtocolChangedException, ConcurrentTransactionException) [docs.delta.io/latest/concurrency-control]. Atomicity of "create version N" is supplied by the storage LogStore, historically the S3DynamoDBLogStore and, since S3 added conditional writes in 2024, increasingly a conditional-write path [delta.io/blog/2022-05-18-multi-cluster-writes-to-delta-lake-storage-in-s3; delta-io/delta#3596].

This sharpens the thesis and corrects one sentence. The claim that "concurrent writers are a property of the catalog" is true for Iceberg on a governed REST catalog (the catalog performs the atomic metadata-pointer compare-and-set, our demoted Lakekeeper binding) and for Delta ONLY under catalog-managed tables with Coordinated Commits (the UCCommitCoordinatorClient, gated by delta.feature.catalogManaged), which apply to managed tables and are not available for external Delta on UC OSS 0.5.0 [unitycatalog.io managed-tables; delta.io/blog catalog-managed-tables]. On our chosen binding, same-table write contention lives in Delta and storage, not in the catalog. What genuinely moves onto the catalog is the metadata, authorization, and credential-vend path.

### Concurrency reduces to table topology, and Model C already supplies the mitigation

Two regimes decide behavior at thousands of agents, chosen by table layout rather than server count. When agents write their own tenant's tables, they share no transaction log and never conflict at any scale; the ceiling is S3 request rate on the hot log prefix and metadata read amplification as logs grow. When agents share one hot table (an audit sink, a common dimension), only one writer wins each version and the losers re-read a longer log and retry, so effective throughput collapses toward serial regardless of fleet size, and Delta raises ConcurrentAppendException even for logically disjoint appends into the same partition unless partition predicates are passed. The mitigation is therefore partition plus table-per-tenant to keep conflict domains disjoint, which Model C's per-tenant layout provides for free, and treating any shared hot table as the scaling hazard.

### A correctness prerequisite the tenancy model itself creates

A few shared Connect servers are several JVMs. Delta's default S3SingleDriverLogStore serializes writers only within one JVM, so two shared servers writing the same tenant's external Delta table on S3 can both believe they wrote version N and silently lose a commit rather than raise a clean conflict. This is a correctness bug, not a throughput tax, and it appears only once more than one server writes the same table. No Delta LogStore is configured in the repository today; every DynamoDB reference is Terraform state locking. Shared servers therefore REQUIRE a multi-JVM-safe LogStore (S3DynamoDBLogStore, or the emerging S3 conditional-write LogStore) before the substrate is even correct. We name this as the first thing to wire and cite, ahead of any throughput number.

### Authorization and vending are the fleet chokepoints for this binding

Every job performs loadTable, a per-principal grant check, and a per-request sts:AssumeRole vend. UC OSS authorizes with JCasbin, an embedded RBAC library whose entire model lives in the UC server JVM; the default Enforcer is not thread-safe, so the contention point at scale is the management plane, every tenant onboard or grant change is a policy reload that must serialize against in-flight checks, not the decision itself. We already record JCasbin as single and non-thread-safe under concurrent policy reload (verified against v0.5.0 source); this promotes that fact from an authorization-maturity footnote to a named scale property. The alternative, Lakekeeper's OpenFGA, is a scale-out Zanzibar-lineage service, so the choice is embedded-library-simplicity versus a scale-out authorization plane, not secure versus insecure; OpenFGA's own frontier is that a low-latency cache mode can serve a stale allow just after a revoke.

The vend has a hard ceiling: the AWS STS default request-rate quota per account per region, shared across STS operations [AWS IAM and STS quotas]. The "no standing credential, vend per request" posture is the safest custody stance and simultaneously the worst for this ceiling; at thousands of agents the two collide. Cross-account vend concentrates the entire fleet's quota onto one account, because only the account making the AssumeRole request counts, so a custodian vending cross-account into tenant accounts funnels every tenant through a single chokepoint regardless of tenant count. The multi-cloud pool sharpens this: STS is the AWS leg only; an on-prem or Databricks member vends by a different mechanism, so the fleet credential plane is heterogeneous and the STS ceiling is one substrate's constraint among several the custodian must juggle. Identity granularity diverges the same way: UC OSS maps the OIDC email claim to a pre-provisioned user and has no JIT principal provisioning, so catalog identity is tenant-grained and static while orchestration identity is agent-grained and ephemeral. The realistic fleet shape is coarse pre-provisioned tenant principals in the catalog with fine-grained per-agent identity pinned above it (our interceptor and sandbox_<principal> convention).

### Federation is structurally mandatory and entirely open

Because the pool spans AWS, on-prem, and Databricks, it is physically multiple catalogs, not one. A tenant's identity, grants, and namespace must stay consistent across substrates, and UC OSS has no mature foreign-catalog or federation story (Lakehouse Federation is a Databricks-managed feature). At scale the failure modes are divergent grant state across catalogs, no single per-tenant view, cross-catalog vend trust, and split-brain authorization. This is the largest unexamined hole and the most defensible honest frontier the tenancy model points at.

### Where this hands off to §4, and why custody is load-bearing

The STS ceiling has one architecturally correct lever, and it is a custody function, not a catalog function: Omnigent holding the vended credential per (principal, prefix) and reusing it within TTL turns STS load from linear-in-requests into linear-in-active-(principal, prefix, TTL-window)-tuples and coalesces concurrent same-prefix requests into one vend. Custody is also where platform-owned commit retry and backoff belong, which matters because untrusted agents supply no backoff discipline and would otherwise turn a hot table into an S3 SlowDown storm on the shared log prefix. This is consistent with the paradigm result: because SDP agents emit only inert declarative transforms and never agent code (§2), the platform, not the agent, owns commit, retry, and vend-cache policy. But custody relieves the ceiling only by concentrating three things: the live downscoped credential material (blast radius), the principal-to-credential binding (moved off the proven catalog gate onto the custodian), and any cross-account cross-substrate quota chokepoint. The safest posture (never standing) is the worst for the quota, and the fix (cache to TTL) reintroduces a bounded standing credential. That tension is the §3-to-§4 seam, and it is why custody is load-bearing rather than plumbing.

### Honest scoping, and the evaluation this section defers

None of the scale behavior above is exercised by the current two-tenant sequential proofs, so all catalog-scale claims are, as of this writing, design property or reasoned frontier, not demonstrated results, in keeping with this repository's standard that every measurement be EKS-native. The minimum credible closing experiment is cheap and high-value and is assigned to the SP4.2 fleet study, not claimed here: (1) configure and cite a multi-JVM-safe Delta LogStore; (2) run N concurrent writers on one hot Delta table versus N on per-tenant tables, on EKS, reporting conflict and retry rate and commits per second for both; (3) exercise vend-QPS against the STS ceiling and the cross-account concentration; and (4) measure the JCasbin authorizer under concurrent grant reload. The §3.4 Frontier box should also split catalog-tier scale (commit model, metadata and authz and vend contention, federation, horizontal catalog replicas over Postgres) from the compute-tier scale (many tenants plus node autoscaling) it currently conflates, because the tenancy thesis points at the former.

## Per-facet investigations

### concurrent-writers
**Findings:** MECHANISM ON OUR SUBSTRATE (UC OSS 0.5.0 + external Delta tables, UCSingleCatalog). Concurrent-writer safety is NOT owned by the catalog on this binding. UC OSS 0.5.0 registers external Delta tables as pointers to an S3 prefix; the actual commit goes through Delta's own transaction log (_delta_log) and its LogStore, not through UC. So for concurrency purposes UC here is a pure metastore. Two independent commit primitives are in play:

(1) DELTA OPTIMISTIC CONCURRENCY CONTROL (per-table). Delta commits in three stages: read latest snapshot, write new data files, then validate-and-commit by attempting to create the next log version file (docs.delta.io/latest/concurrency-control.html). Conflicts are detected only at commit and raised as typed exceptions: ConcurrentAppendException (files added to a partition another op read), ConcurrentDeleteReadException / ConcurrentDeleteDeleteException (a file you read/deleted was rewritten by a concurrent DELETE/UPDATE/MERGE/compaction), MetadataChangedException (concurrent metadata/schema change), ProtocolChangedException, ConcurrentTransactionException (shared streaming checkpoint). Conflict matrix: blind INSERT/appends never conflict with each other; UPDATE/DELETE/MERGE conflict with each other and with appends into the same partition. The documented mitigation is DISJOINT CONFLICT DOMAINS: partition the table on the columns your predicates filter on and pass explicit partition filters, so concurrent writers touch disjoint file sets and pass validation. Retry of a logical conflict is an APPLICATION-level concern (Delta guarantees no partial writes, so retry is safe), but each retry re-reads the tail of _delta_log, so retry cost grows with log length and writer count.

(2) THE STORAGE-LEVEL ATOMIC-COMMIT PRIMITIVE (the real fleet-scale floor). Delta's per-table OCC only works if 'create version N.json' is atomic. On S3 that was historically impossible (no atomic put-if-absent), so multi-writer-across-JVMs on S3 requires the S3DynamoDBLogStore, which uses a DynamoDB conditional PutItem to arbitrate which writer 'wins' version N (delta.io/blog/2022-05-18-multi-cluster-writes-to-delta-lake-storage-in-s3). Since Aug 2024 S3 supports conditional writes (If-None-Match), and Delta is moving toward a DynamoDB-free LogStore; delta-rs already uses native S3 conditional writes (>=0.23.0), but the Delta-Spark JVM conditional-write LogStore is still an open item (delta-io/delta#3596). The default S3SingleDriverLogStore only serializes writers WITHIN ONE JVM. CRITICAL FOR MODEL C: a FEW SHARED Connect servers = several JVMs. Two Connect servers writing the SAME tenant's Delta table on S3 with the default LogStore and no DynamoDB/conditional-write LogStore do not get a clean conflict exception; they can BOTH believe they wrote version N and silently lose a commit / corrupt the log. This is a correctness bug, not a throughput tax.

(3) THE CATALOG-COORDINATED ALTERNATIVE we do NOT use. Delta 4.2+ adds Coordinated Commits / catalog-managed tables: a CommitCoordinatorClient (UCCommitCoordinatorClient) lets the catalog broker commits, staging to _delta_log/_staged_commits then ratifying, so conflict detection and the latest-metadata pointer live in the catalog rather than the filesystem (delta.io/blog/2026-02-02-delta-catalog-managed-tables; DeepWiki delta 4.4). This went GA on Databricks UC and enables multi-table atomic commits and lower metadata latency. BUT it applies to UC MANAGED tables, not external ones; UC OSS is only beginning to land the open Delta commit APIs (Delta 4.3, delta.io/blog/2026-06-22-delta-4-3-release). Our binding is EXTERNAL Delta on UC OSS 0.5.0, which predates and does not use this. So on our stack the catalog is deliberately OUT of the commit path.

ICEBERG CONTRAST (the demoted Lakekeeper path). Same optimistic model, different atomic primitive: each writer writes a new metadata.json and asks the CATALOG to atomically swap the table's current-metadata pointer (compare-and-set); if another writer swapped first, it re-reads the new snapshot, re-plans, and retries (iceberg.apache.org/spec; AWS Glue concurrent-write blog). So Iceberg's atomic commit lives in the catalog by design (Lakekeeper/Glue do the CAS), whereas external-Delta's lives in the storage LogStore. Iceberg exposes explicit tunables (commit.retry.num-retries default 4, commit.retry.min/max-wait-ms) and the default 4 is widely reported as too low under many concurrent writers, producing ICEBERG_COMMIT_ERROR. Net: Lakekeeper/Iceberg already puts commit arbitration on the catalog (validating the author's thesis for that path); external-Delta/UC-OSS puts it on storage (so the thesis needs a caveat for our chosen path).

**Scale behavior:** Two regimes decide behavior at thousands of agents, and which one you are in is set by TABLE TOPOLOGY, not by the number of Connect servers.

DISJOINT-TABLE REGIME (the good case, and what Model C's per-tenant layout buys). If agents mostly write their OWN tenant's tables/prefixes, cross-tenant writers never share a _delta_log, so they NEVER conflict, no matter how many agents. Conflicts and retries are intra-tenant only. The ceiling here is not OCC at all, it is (a) S3 request rate on the hot _delta_log prefix (~3,500 PUT/s per prefix) and metadata read amplification as logs grow, and (b) the DynamoDB LogStore hot-partition throughput if used for multi-JVM safety. This scales to a large fleet gracefully. Table-per-tenant plus partitioning is therefore the primary concurrency mitigation, and it falls out of the tenancy model for free.

HOTSPOT-TABLE REGIME (the failure mode). A single shared/hot table (a common dimension, an audit/events sink, a leaderboard) with N agents appending concurrently degrades superlinearly. Only ONE writer wins each version; every other writer takes a ConcurrentAppendException (Delta) or CAS failure (Iceberg), re-reads a now-longer log, and retries, i.e. a thundering herd whose per-retry cost rises with both N and log length. Effective commit throughput collapses toward serial: ~1 commit per commit-latency, which under contention is seconds, so aggregate write rate to that table is a small constant regardless of fleet size. Delta append-append is especially treacherous because two logically-disjoint appends to the SAME partition still raise ConcurrentAppendException unless partition predicates are supplied; agents that do not pass partition filters manufacture false conflicts. Iceberg's default commit.retry.num-retries=4 means the losers simply error out (ICEBERG_COMMIT_ERROR) rather than starving.

THREE FLEET-SCALE HAZARDS SPECIFIC TO OUR MODEL C. ### authz-vend-at-scale — per-principal authorization (L4) + credential vending (L5) at fleet scale
**Findings:** Two distinct mechanisms sit behind L4/L5, and they scale for opposite reasons.

L4 AUTHORIZATION ENGINES. UC OSS 0.5.0 authorizes via JCasbin, an *embedded* RBAC library: the whole policy model (users, roles, catalog/schema/table securables, grants) is loaded into the UC server JVM's memory and Enforce() is a matcher evaluation over that in-memory set. Our proof log confirms the vend endpoint is gated by @AuthorizeExpression(VEND_TABLE_CREDENTIAL) (READ->SELECT, READ_WRITE->SELECT+MODIFY) and that the authz decision fires BEFORE any STS call (Part A: cross-tenant vend -> HTTP 403 PERMISSION_DENIED, no AssumeRole attempted). JCasbin's default Enforcer is NOT thread-safe; thread safety requires SyncedEnforcer, which guards reads/writes with a lock (jcasbin README/DeepWiki). Our own source read recorded exactly this: "JCasbin is single/non-thread-safe under concurrent policy reload." By contrast Lakekeeper's OpenFGA is a Zanzibar-lineage *distributed service* with its own datastore, relationship tuples instead of a monolithic in-memory model, caching, and tunable consistency modes (MINIMIZE_LATENCY vs HIGHER_CONSISTENCY). OpenFGA is documented at ~1M RPS over ~100B tuples with sub-ms p99 when cached (openfga.dev production guidance; Zanzibar/Leopard ~1.5M QPS, sub-150us median). So the L4 choice is not security-equivalent-vs-not; it is embedded-library-simplicity (JCasbin, one process, one Postgres) vs scale-out-authz-plane (OpenFGA, separate horizontally-scalable service).

L5 CREDENTIAL VEND. Both catalogs mint short-lived cloud creds the same way: sts:AssumeRole against a broad vending role, narrowed by an inline session policy to the tenant's key-prefix, pinned with sts:ExternalId (confused-deputy). UC: AwsCredentialVendor issues AssumeRoleRequest.externalId() and AwsPolicyGenerator builds the prefix-scoped inline policy (verified in v0.5.0 source per our log; documented in UC internals). Lakekeeper: IRSA catalog identity assumes lakekeeper-vending, external-id-gated, downscoped per warehouse (lakekeeper-vending.tf lines 76-129; max_session_duration=3600, i.e. 1h TTL). Live proof on both: UC 6/6 (real ASIA* downscoped creds, cross-tenant PERMISSION_DENIED both directions, replayed cred -> S3 AccessDenied); Lakekeeper the same, plus CloudTrail showing the fleet IRSA role makes zero data calls.

IDENTITY MODEL. UC OSS maps the OIDC email claim to a PRE-PROVISIONED UC user; there is no default JIT provisioning (our gap table, line 647; UC docs describe permanent catalog access + JIT-to-engine only for the *credential*, not the *principal*). Databricks-managed UC gates vending on EXTERNAL USE SCHEMA privilege, an OSS-absent managed feature (Databricks docs); OSS uses object-level grants only.

**Scale behavior:** L4 at fleet scale. JCasbin's per-call Enforce is cheap (microseconds) but three things bite at thousands of principals: (1) the in-memory model is O(principals x securables x grants) inside ONE JVM, no sharding, no distributed store; (2) every tenant onboard / grant change is a policy reload, and reloads on the non-thread-safe Enforcer must be serialized against in-flight checks (SyncedEnforcer's lock) or race — so the *management plane* (grant churn), not the decision, is the contention point; (3) UC is a single server + Postgres, so authz throughput is bounded by one process. OpenFGA sidesteps all three by construction (scale-out service, tuple store, cache) but trades in the caching/consistency knob: MINIMIZE_LATENCY can serve a stale allow just after a revoke, which for a security boundary is the wrong-direction staleness you must reason about.

L5 at fleet scale is governed by ONE hard number: AWS STS default quota is 600 requests/second per account per region, SHARED across all STS ops (AssumeRole, GetCallerIdentity, etc.) (AWS IAM/STS quotas doc; re:Post throttling guidance). Model C's stated posture is "no session ever holds a standing credential" = vend per request, short TTL. At fleet scale that posture directly maximizes STS QPS: every agent request that touches storage is one AssumeRole. Thousands of agents each doing multiple table ops trivially approach 600/s. Two amplifiers specific to our design: (a) cross-account vend concentrates the quota — "only the account making the AssumeRole request impacts the STS quota," so if the vending role is in a tenant account and the catalog/custodian calls cross-account, ALL fleet vends bill to the single catalog account's 600/s regardless of tenant count; (b) the 1h max_session_duration means TTL churn is bounded but if you never reuse, you pay full STS cost per request. Mitigations are documented (regional STS endpoints get separate quotas; quota increase by support ticket) but the architecturally correct one is CACHING/REUSING creds within TTL — which is precisely a *custody* function, not a catalog function. Multi-cloud sharpens this: STS is the AWS leg only; an on-prem or Databricks pool member vends via a different mechanism (Azure SAS, GCP tokens, or UC's own vend), so the fleet's credential plane is heterogeneous and the 600/s ceiling is one substrate's constraint among several the custodian must juggle.

Identity at scale: pre-provisioning is an onboarding wall. If per-agent identities are ephemeral (per-task), you cannot mint a UC user + grants inline before the agent works. The realistic fleet shape is therefore coarse pre-provisioned TENANT principals in the catalog + fine-grained per-agent identity pinned above it (our PrincipalPinningInterceptor + sandbox_<principal> convention), with the agent->tenant map enforced outside UC.

HOW §4/OMNIGENT CUSTODY RELIEVES vs CONCENTRATES. Custody is the natural cache/coalesce point: Omnigent holding the vended credential per (principal, prefix) and reusing it within TTL turns STS load from linear-in-requests into linear-in-active-(principal,prefix,TTL-window)-tuples — the single biggest lever on the 600/s ceiling, and it also coalesces concurrent same-prefix requests into one vend. So custody RELIEVES the STS bottleneck. But it CONCENTRATES three things: (1) credential material — the custodian becomes the one high-value store holding live downscoped creds (blast radius); (2) the principal<->credential binding must now be enforced by the custodian, not the catalog, moving a security-critical check off the proven catalog gate; (3) any cross-account cross-substrate quota chokepoint now lives in one service. The tension is exact and paper-worthy: the safest custody posture (vend-per-request, never standing) is the WORST posture for STS quota, and the fix (cache to TTL in custody) reintroduces a bounded standing credential and concentrates trust.

**Our state/gap:** PROVEN IN-SYSTEM: L4+L5 correctness on both catalogs at N=2 tenants.
