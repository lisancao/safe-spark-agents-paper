# §3 platform rework: gap map + Lakekeeper→UC-OSS-native swap plan (2026-07-16)

Assessment-driven (5-lens platform audit). Principle: **running locally disproves the thesis** — every proof must be EKS-native.

## Current state

The platform hides two maturity tiers behind PAPER.md §3's uniform "all five layers run on live EKS" prose. WHAT IS EKS-NATIVE AND RUNNING (catalog-independent, survives the swap): L2 token custody and L3 pod/JVM execution isolation on ssa-spark-eks (ns spark). Live image ssa-spark/spark-connect:4.1.2-iceberg1.11.0 bakes the real fail-closed PrincipalPinningInterceptor (Dockerfile lines 65-78, entrypoint.sh line 22), pins user_id to x-connect-principal, real distributed executor pods with per-tenant tenant=<T> labels and distinct IPs, shared PSK custody (Secret spark-connect-psk) live. Verified 2026-07-14 (proof_2026-07-14_live_isolation.log) and 2026-07-16 (H3_EKS_SWEEP.md). WHAT IS EKS-NATIVE BUT LAKEKEEPER-COUPLED (must be reworked, not reused): the whole catalog/storage plane. gen_connect_server.py:36-43 hardwires the Iceberg-REST 'lk' catalog (type=rest, uri=lakekeeper-authz:8181/catalog, Bearer <Lakekeeper JWT>, X-Iceberg-Access-Delegation=vended-credentials); L4 is OpenFGA/Zanzibar (404-deny semantics, 00-idp-openfga.yaml + 10-lakekeeper-authz.yaml, Lakekeeper v0.13.1); L5 is deploy/eks/terraform/lakekeeper-vending.tf (lakekeeper_catalog IRSA assumes lakekeeper_vending, operator-set sts:ExternalId) + spike/eks/warehouse-tenant_*.aws.json. Live catalog is 'lk_a'; the old iceberg JDBC catalog is DEAD but stale config still lingers in connect/base/configmap-env.yaml + entrypoint.sh. WHAT IS CONFIGURED/PROVEN-ONLY (never composed live): L1 mTLS ingress. gateway-envoy.yaml + 20-connect-gateway.yaml + service-mtls.yaml implement SAN-based per-principal routing, proven only in isolation (proof_2026-07-10_ingress_routing.log, ephemeral servers + local proxy); the live H3 run BYPASSED the gateway (port-forward to pod:15002 + hand-injected header + PSK). The deployed Connect CA is rotated and its private key is NOT on host (fingerprint 48:A1:81:59 deployed vs 8A:B3:F3:29 on-disk), so client certs cannot currently be minted. L3 NetworkPolicy is INERT (task #26 pending, VPC CNI ENABLE_NETWORK_POLICY off). WHAT IS SPIKE-ONLY / OFF-EKS (the required target, least-built): Unity Catalog OSS. The only committed artifacts are deploy/auth/uc/README.md + spark-defaults.uc.conf (the WRONG fleet single-token binding whose own README says per-user grants are not enforced). The strong per-principal 6/6 result (proof_2026-07-10_uc_vending.log) ran entirely off-cluster: UC v0.5.0-SNAPSHOT from-source (pinned commit 0f14452 + two patched authz bugs), H2 backend, ambient AWS session (NOT IRSA), Spark 4.0.1 with LOCAL-THREAD executors, ephemeral torn-down AWS resources; the proof scripts lived in scratchpad/ and are GONE. Zero UC k8s manifests, terraform, per-tenant generator, or EKS proof exist. GITOPS GATE/RECONCILE: reconcile.py is catalog- and endpoint-agnostic (one SPARK_REMOTE swap), but both shipped workflows target runner-local sc://localhost:15055 on GitHub-hosted runners; PRODUCTION_EKS.md is "documented, NOT enabled," stale (cites plaintext :15002 not governed :15009, omits the mTLS-cert+PSK requirement). NET: L1/L2/L3 are EKS-native but carry the wrong catalog's token; UC-native L4/L5 exist only as an off-cluster spike; there is ZERO evidence of UC-OSS-native L1-L5 composed on the live cluster.

## Correctness prerequisite (from the catalog investigation, 2026-07-16)

**S3DynamoDBLogStore (or the S3 conditional-write LogStore) is REQUIRED before any shared-server (Model C) Delta write claim — this is correctness, not performance.**

- Model C = a few shared Connect servers = multiple JVMs. Delta's default `S3SingleDriverLogStore` serializes concurrent writers ONLY within a single JVM.
- Two shared servers writing the same tenant's external Delta table on S3 with the default LogStore can each believe they created log version N -> silent lost commit / log corruption.
- The repo ships NO Delta LogStore config today (the unsafe default). (Every DynamoDB reference in the repo is Terraform state-locking, unrelated.)
- Fix (off-the-shelf): add `delta-storage-s3-dynamodb` to the UC-native image jars, provision a DynamoDB table, and set `spark.io.delta.storage.S3DynamoDBLogStore` (+ table/region) in the Connect server config. Alternatively the emerging S3 conditional-write LogStore for Delta-Spark JVM (delta-io/delta#3596) once released; delta-rs already uses S3 If-None-Match since 0.23.0.
- Catalog note: on UC-OSS + external Delta the catalog is a metadata authority, NOT the commit arbiter (commits are Delta OCC + the LogStore). Catalog-as-commit-arbiter only applies to Iceberg-on-REST (demoted Lakekeeper) or UC managed tables + Coordinated Commits (not chosen). Full analysis: paper/notes/catalog_dimension_findings.md.

Sequence: fold this into "Build and push the UC-OSS EKS image" (step ~3) so the LogStore jar + config ship with the image, and the DynamoDB table into the UC terraform.
